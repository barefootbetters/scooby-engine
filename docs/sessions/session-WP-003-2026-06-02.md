3. **First pass — region characterization (high-level).** Read the full 1,045,046-byte range `0x0F48`–`0x10017E` into memory (Python `Path.read_bytes` + slice, or any hex viewer's go-to-offset + select). Compute:
   - Byte-value histogram — long runs of `0x00` indicate padding; uniform distribution suggests compressed or structured data
   - First non-zero byte after `0x0F48` (the structure-start offset)
   - Last non-zero byte before `0x10017E` (the structure-end offset)
   - Any periodic structure — try strides of 3, 4, 6, 768, 1024 (palette candidates) and 8, 12, 16 (entry-table candidates)

4. **Targeted hypothesis checks** in order, each citing offset + hex window + verdict:
   - **Palette hypothesis A — 256-entry RGB (768 bytes):** is there a contiguous 768-byte run at any offset in the region where bytes look like color values (no extreme outliers, plausible gradients)?
   - **Palette hypothesis B — 256-entry RGBA (1024 bytes):** same check at stride 4
   - **Palette hypothesis C — multiple palettes:** does the region contain multiple 768-byte or 1024-byte blocks at regular strides?
   - **Name table hypothesis:** does the region contain ASCII strings ≥ 6 chars? If yes, sample a handful and cross-check against `tools/exes/showdown/strings-ansi.txt` (WP-007 dump). A candidate appearing in both is a strong signal; appearing only in TGIFILE.ART is a weaker signal (the name table might be data-local rather than code-referenced).
   - **Frame metadata hypothesis:** is there entry-shaped data (fixed-stride records) that could be per-asset metadata (width, height, palette index, frame count)?
