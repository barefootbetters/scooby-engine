## After execution ‚Äî lessons learned

**Lesson ‚Äî pre-flight did not anticipate that the format-doc itself might be wrong.** The pre-flight's ¬ßRepo + disc state row "TGIFILE.ART cached ‚Ä¶ SHA matches" is necessary but not sufficient: the cached binary matched the SHA, yet the existing [tgifile-art.md ¬ßHeader layout](../formats/tgifile-art.md#header-layout) table had `asset_count` at the wrong offset (`0x011C` vs the actual `0x0118`) ‚Äî a +4 shift across the whole header. Step 1 of the session prompt's "sanity-check the cached binary" would have failed (`asset_count @0x011C` decodes as 1,048,958, not 453) and triggered a STOP, except step 2 explicitly anticipated outcome (c) "the asset entry count is actually at `0x0118` not `0x011C`". The session prompt's design saved the run by sequencing step 2 right after step 1.

For future pre-flights: when a WP's main artifact is an *update* to an existing format-doc, the pre-flight ¬ßRepo + disc state should include a "cross-check the binary against the doc" row that exercises at least one non-trivial offset cited in the doc. Catching spec drift at the pre-flight gate is cheaper than catching it mid-session via a step-1 mismatch.
